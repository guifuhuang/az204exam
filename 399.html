您正在使用 Azure Event Grid 实现一个应用程序，以向客户推送近实时信息。

您有以下要求：

  • 必须向数千名客户发送事件，这些事件包含数百种不同类型
  • 事件在处理前必须按事件类型进行过滤
  • 必须使用 Microsoft Entra ID 处理身份验证和授权
  • 事件必须发布到单个端点

您需要实现 Azure Event Grid。

解决方案： 将事件发布到系统主题（system topic）。为每个客户创建一个事件订阅。

该解决方案是否满足目标？

答案与解析

正确答案: B. No

解析：

该解决方案无法满足所有要求，主要原因如下：

1. 系统主题的限制

系统主题（system topic）是由Azure服务（如存储账户、订阅等）自动创建的，主要用于Azure服务自身的事件。它不适合用于向外部客户分发自定义事件，特别是当事件类型多样且需要精细控制时。

2. 订阅数量限制

Azure Event Grid 对每个主题的订阅数量有限制（通常每个主题最多支持500个订阅）。题目要求支持数千名客户，这意味着需要创建数千个事件订阅，这远远超出了系统主题的订阅限制。

3. 端点要求不满足

题目要求事件必须发布到单个端点。但解决方案中为每个客户创建单独的事件订阅，每个订阅都需要配置不同的端点，这违反了"单个端点"的要求。

4. 身份验证问题

系统主题使用Azure资源的内置身份验证机制，难以与Microsoft Entra ID集成来实现对外部客户的细粒度访问控制。

正确的解决方案：事件域（Event Domain）

要实现所有要求，应使用事件域（Event Domain）：

  • 事件域作为管理大量主题的顶级资源，支持多租户场景
  • 可以向单个域端点发布事件
  • 在域内为每个客户创建域主题（domain topic）
  • 为每个域主题创建事件订阅，并按事件类型过滤
  • 支持使用Microsoft Entra ID进行身份验证和授权

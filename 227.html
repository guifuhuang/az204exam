QUESTION NO: 227

Scenario:

You are developing an Azure Service application that processes queue data when it receives a message from a mobile application. Messages may not be sent to the service consistently.

Requirements:

  • Queue size must not grow larger than 80 gigabytes (GB)
  • Use first-in-first-out (FIFO) ordering of messages
  • Minimize Azure costs

Solution:

Use the .Net API to add a message to an Azure Service Bus Queue from the mobile application. Create an Azure Windows VM that is triggered from Azure Service Bus Queue.

Question:

Does the solution meet the goal?

Answer:

B. No

Explanation:

该解决方案不符合要求，原因如下：

  1. 成本问题：
    • 使用Azure Windows VM会增加计算成本，特别是当消息处理不频繁时[12](@ref)[13](@ref)
    • 更经济的解决方案是使用Azure Functions，它采用无服务器架构，按执行计费[6](@ref)[7](@ref)
  2. 架构问题：
    • Azure Service Bus Queue原生不支持直接触发Windows VM[3](@ref)[5](@ref)
    • 正确的触发方式应使用Azure Functions或Logic Apps[6](@ref)[8](@ref)
  3. 技术实现问题：
    • 虽然Service Bus Queue支持FIFO排序[1](@ref)[3](@ref)，但使用VM作为处理器可能无法保证严格的顺序处理
    • VM需要手动配置和管理，增加了运维复杂度[12](@ref)[14](@ref)

推荐解决方案：

  1. 使用Azure Service Bus队列（支持FIFO和80GB容量）[3](@ref)[5](@ref)
  2. 采用Azure Functions作为消息处理器（成本更低）[6](@ref)[7](@ref)
  3. 配置Service Bus触发器自动处理消息[8](@ref)

关键区别：

方案 成本 FIFO支持 运维复杂度
VM方案 高（按小时计费）[12](@ref) 可能中断 高
Functions方案 低（按执行计费）[6](@ref) 完整支持 低
